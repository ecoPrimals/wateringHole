deployment to a universal substrate artifact: any OS, any architecture, any
filesystem. This handoff documents the patterns, decisions, and evolution

## For biomeOS Team

### ColdSpore Marker Evolution

The `.biomeos-spore` marker and `biomeOS/tower.toml` are hand-written in
`prepare-usb.sh`. These should be generated by biomeOS's `create_livespore.sh`.

### Patterns to Absorb

1. **`detect_os()` in `_lib.sh`**: Identifies Linux, Darwin, and Windows-shell
   (MINGW/MSYS/CYGWIN). This pattern should live in a biomeOS shared library
   rather than being duplicated across springs.

2. **Container dispatch (`container_exec()`)**: Full pattern for transparent
   Docker/Podman delegation with GPU passthrough flags. Includes image loading,
   runtime detection, and volume mounts for results persistence.

3. **`prepare-usb.sh` dual-mode**: Supports ext4 (Linux-native, preserves
   permissions) and exFAT (universal readability, requires tmpdir fallback).
   This should evolve into `create_livespore.sh` having a `--data-spore` mode
   alongside the current boot-spore mode.

4. **`graphs/hotspring_deploy.toml`**: Currently does not model the container
   dispatch path. The deploy graph needs container awareness as a node type.

### liveSpore.json Schema

The `liveSpore.json` manifest now includes:

```json
{
  "architectures": { "x86_64": {...}, "aarch64": {...} },
  "cross_platform": { "linux_native": true, "windows_wsl2": true, ... },
  "deployment_strategy": { "linux_x86_64": "Direct execution", ... },
  "validated_substrates": ["x86_64 Ubuntu", "aarch64 Ubuntu", ...]
}
```

This should converge with the `genomeBin/manifest.toml` format.

## For toadStool Team

### Dual-Binary Strategy

Static musl binaries cannot `dlopen("libvulkan.so.1")`. The hotSpring
artifact ships two binary variants per architecture:

- `bin/<arch>/static/<name>` — musl, CPU-only, zero dynamic deps
- `bin/<arch>/gpu/<name>` — glibc, GPU-capable (needs libc + libm + libgcc_s)

**Evolution target**: toadStool's substrate detection should detect
"calling binary is static-pie, Vulkan dlopen will fail" and suggest the
GPU binary alternative. This logic currently lives in `_lib.sh`'s
`detect_gpu()` and `resolve_binary()`.

### tmpdir Fallback Pattern

When `chmod +x` fails on the resolved binary (exFAT, NTFS, NFS noexec),
the binary is copied to `$TMPDIR`, made executable there, and executed
from the tmp location. This pattern should be absorbed into toadStool's
binary resolution.

### aarch64 Static Target

`BARRACUDA_PRIMAL_BUDDING.md` should note that `aarch64-unknown-linux-musl`
is now a validated cross-compile target for CPU-only binaries. GPU binaries
on aarch64 require native ARM hardware with Vulkan.

## For primalSpring Team

### ecoBin UniBin Reference Pattern

`./hotspring` is the reference implementation of the ecoBin UniBin pattern
for springs. Key design decisions:

1. **Shell wrapper, not Rust mega-binary**: 19 Rust binaries dispatched via
   shell. Binary size (8MB each vs 50+ merged), parallel compilation, and
   feature isolation justified the choice.

2. **`_lib.sh` shared library**: Four functions (integrity_check, detect_arch,
   detect_gpu, resolve_binary) eliminate ~150 lines of duplication across
   6+ scripts.

3. **Arch-first layout**: `bin/<arch>/{static,gpu}/<name>` matches
   benchScale's `BinaryResolver` convention and the physical decision tree
   ("what machine?" then "what can it do?").

4. **Legacy symlinks**: All old entry points (`./run`, `./benchmark`,
   `./chuna-engine`, `./deploy-nucleus`) forward to `./hotspring <subcommand>`.
   Backward compatibility is preserved without code duplication.

### CHECKSUMS Pipeline

`build-guidestone.sh` generates SHA-256 checksums for all binaries, scripts,
and reference data. The integrity check runs before every execution via
`integrity_check()` in `_lib.sh`.

### qemu-user binfmt Pattern

Cross-architecture Docker validation requires qemu-user binfmt registration:

```bash
docker run --rm --privileged multiarch/qemu-user-static --reset -p yes
```

This enables `docker run --platform linux/arm64 arm64v8/ubuntu:22.04` on
x86_64 hosts. The registration persists until reboot.

### agentReagents VM Template

`sort-after/agentReagents/templates/springs/gate-hotspring-aarch64.yaml`
provides a full ARM64 VM validation gate:

- Ubuntu 24.04 ARM64 cloud image
- Copies hotSpring artifact
- Runs `HOTSPRING_NO_GPU=1 ./hotspring validate`
- Verifies `uname -m` is `aarch64` and all checks pass

## For All Teams

### Container Deployment as Universal Fallback

The OCI container image pattern (`Dockerfile` + `build-container.sh` +
`container/hotspring-guidestone.tar`) should be adopted by any spring that
ships a validation artifact:

1. **Base**: Ubuntu 22.04 (or appropriate minimal base)
2. **Deps**: Only runtime libraries (e.g., `libvulkan1` for GPU)
3. **Content**: Full artifact directory at `/opt/validation`
4. **Entry**: Same `./hotspring` (or `./springname`) entry point
5. **Export**: `docker save` to `.tar` for offline transport

The container provides the universal fallback when native execution is not
possible (macOS, Windows without WSL2, restricted HPC, NFS noexec mounts).

### Cross-OS Launcher Pattern

- **Windows**: `springname.bat` checks WSL2 → Docker Desktop → prints instructions
- **macOS/Linux non-native**: Entry point script detects OS via `uname -s`,
  dispatches to container if not Linux
- **Linux native**: Direct binary execution with arch + GPU auto-detection

### Filesystem Compatibility Matrix

| Filesystem | Execute Bit | Strategy |
|------------|-------------|----------|
| ext4 | Native | Direct execution |
| exFAT | No | tmpdir fallback (transparent to user) |
| NTFS (WSL2) | Emulated | WSL2 handles permissions |
| NFS noexec | Blocked | tmpdir fallback |
| FAT32 | No | tmpdir fallback (or avoid — only use for data) |

## Known Gaps and Evolution Targets

1. **toadStool ColdSpore socket**: GPU binary logs noise when no biomeOS
   socket exists. Should detect `BIOMEOS_DEPLOYMENT_MODE=cold` and skip.

2. **ARM GPU binaries**: `bin/aarch64/gpu/` not possible without native ARM
   GPU hardware for Vulkan dlopen. Future: Jetson, Apple Silicon via Asahi.

3. **genomeBin manifest convergence**: `liveSpore.json` and
   `genomeBin/manifest.toml` should share a schema.

4. **primalSpring composition**: Artifact should be producible via
   primalSpring graph execution, not shell scripts.

5. **ILDG guided generation**: `./hotspring generate --ns=16 --nt=32
   --beta=6.0 --configs=100` would make the USB a complete physics tool.

6. **Container deploy graph**: `biomeOS/graphs/hotspring_deploy.toml` does
   not model the container dispatch path.
