## SD-03: GenomeBin Signing (Ed25519)

### Design decision: Ed25519 over sequoia-openpgp

The ecosystem roadmap (plasmidBin PHASE58) explicitly converges on:
> BLAKE3 checksums → **Ed25519 detached signatures** → BearDog-derived keys → Sigstore

`sequoia-openpgp` was rejected because:
1. Default crypto backends use C libraries (Nettle), violating ecoBin "zero C deps"
2. Even "pure Rust" backends require non-trivial feature configuration
3. OpenPGP is heavyweight for the ecosystem's actual needs
4. Ed25519 is the explicit next step in the plasmidBin signing roadmap

### What was implemented

New module: `sourdough-genomebin/src/signing.rs`

**Public API:**
- `generate_keypair() -> (SigningKey, VerifyingKey)` — OS-random Ed25519 keypair
- `sign_file(path, key) -> Signature` — BLAKE3 hash file content, Ed25519 sign the hash
- `verify_file(path, sig, key) -> bool` — re-hash, verify signature
- `write_signature(sig, artifact_path)` — hex-encoded `.sig` sidecar
- `read_signature(artifact_path) -> Signature` — load from sidecar
- `write_verifying_key(key, path)` / `read_verifying_key(path)` — key persistence
- `signature_path_for(artifact_path) -> PathBuf` — derive `.sig` path

**Dependencies added (workspace-level):**
- `ed25519-dalek = { version = "2", features = ["rand_core"] }` — pure Rust Ed25519
- `rand_core = { version = "0.6", features = ["getrandom"] }` — OS randomness

Both are pure Rust with zero C dependencies. `cargo deny check` passes clean.
